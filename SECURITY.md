# 安全与隐私 / Security & Privacy

本项目处理的是**最敏感的一类个人数据**——心理、健康、人际关系。安全设计的第一原则是：**真实数据永不进入公开仓**。

本文档说明项目的隐私威胁模型、隔离机制，以及如何私下披露漏洞与隐私问题。

---

## 一、隐私威胁模型（概述）

我们重点防范的，不是远程攻击者，而是**真实个人数据意外泄露到公开仓**这一最现实的风险。主要威胁面：

1. **误提交**：用户在本地用真实数据填好模板后，误把 `$ME_HOME` 中的档案 commit 进公开仓。
2. **示例污染**：贡献者用真实的人/事/病/钱写"示例"，把真实信息混进 `examples/`。
3. **第三方信息**：档案天然包含他人（家人、同事）的信息，泄露会波及未同意的第三方。
4. **内部信息外溢**：在公司环境使用时，把内部项目代号、内网地址等带进公开内容。
5. **截图与日志**：issue/PR 中粘贴的截图、报错日志里夹带真实路径或内容。

我们**不**承诺防御具有本机完整权限的攻击者——`$ME_HOME` 是明文本地文件，其物理与系统级安全由用户的操作系统与磁盘加密负责。

---

## 二、双仓隔离（核心机制）

本项目采用**框架仓 / 数据仓分离**的设计：

| | 框架（本公开仓） | 用户数据 |
|---|---|---|
| 内容 | 镜头、协议、方法论、模板、虚构示例 | 真实人物/议题/健康档案 |
| 位置 | 本仓（`SKILL.md` / `experts/` / `frameworks/` / `toolkit/` …） | 数据根 `$ME_HOME`（默认 `~/.me`） |
| 是否公开 | ✅ 公开 | ❌ **永不公开、永不入本仓** |

- 框架文件与本仓同级；用户运行时，框架从 `$ME_HOME` **读取/写入**真实数据，但真实数据**不属于**框架仓的版本管理范围。
- 所有真实数据路径在文档与模板中一律写作 `$ME_HOME/...`，**不写绝对真实路径**。
- 通知 / 后台沉淀等本地私有任务属于用户侧，不在公开仓内。

---

## 三、redact-check 门禁

仓库提供 `scripts/redact-check.sh` 作为**提交前的脱敏门禁**：

- 扫描改动，匹配疑似真实个人信息、第三方姓名、内网地址、内部项目代号、电话/邮箱/精确日期等模式；
- 校验示例是否使用了规范占位符（`<...>` / `{{...}}`）；
- **检查未通过的 PR 不予合并**（建议同时配置为 CI 必过项与本地 pre-commit 钩子）。

门禁是**底线而非保证**——它会拦住常见模式，但无法识别所有伪装的真实信息。最终责任在提交者。

---

## 四、私有黑名单

每个用户可在本地维护一份**私有黑名单** `$ME_HOME/.redact-blocklist`，列出绝不能出现在任何输出/提交中的词（真实人名、公司名、项目代号、地名等）：

- 黑名单本身属于敏感信息，**存放于 `$ME_HOME/.redact-blocklist`，本身也不入公开仓**；
- `redact-check.sh` 在本地运行时会读取 `$ME_HOME/.redact-blocklist`，对改动做额外比对；
- 黑名单条目使用真实词触发拦截，因此**绝不要把黑名单内容贴进 issue/PR**。

> 黑名单的结构模板见 `templates/`（模板只含字段结构与占位符，不含任何真实词）。

---

## 五、漏洞与隐私问题的私下披露

如果你发现**安全漏洞**，或注意到**仓库中已混入真实个人信息**，请**不要**公开提交 issue 或 PR（那会进一步放大泄露）。

请通过私下渠道披露：

1. 使用代码托管平台的**私密漏洞上报**功能（如 GitHub Security Advisory 的 "Report a vulnerability"），或
2. 联系维护者的私密联系方式（见仓库主页 `README.md` 中维护者指定的渠道）。

披露时请尽量包含：

- 问题类型（漏洞 / 真实数据泄露 / 其他隐私风险）；
- 受影响的文件或提交（**用占位符描述位置，不要复述泄露的真实内容**）；
- 复现或定位方式。

我们会尽快确认并处理。**若涉及真实数据已进入公开历史，请在私下沟通中说明**，以便评估是否需要重写历史 / 失效相关凭据 / 通知受影响方。
