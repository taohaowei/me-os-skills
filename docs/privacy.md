# 隐私模型

> 本文档说明本框架如何保证你的真实档案数据「永不出仓」，以及当你记录到第三方（家人 / 同事 / 上级）时应遵守的伦理边界。

本框架的第一性原则：**框架是公开的镜头与方法论，数据是私有的真实档案。两者物理隔离，永不混在同一个仓库里。**

---

## 1. 为什么数据永不出仓

助手会逐步沉淀关于你的高度敏感内容：心理状态、家庭关系、职场冲突、健康记录、个人决策。这类数据一旦泄露，伤害不可逆。因此框架的设计目标不是「尽量别泄露」，而是**让泄露在结构上不可能发生**。

实现方式是双仓分离：

| | 公开框架仓（本仓） | 私有数据仓 `$ME_HOME`（默认 `~/.me`） |
|---|---|---|
| 内容 | 镜头 / 协议 / 方法论 / 模板 / 示例 | 你的真实档案、状态、关系、健康、决策记录 |
| 可见性 | 公开开源 | 仅本地，永不推送 |
| 占位符 | 一律 `<...>` / `{{...}}` | 填入真实值 |
| git remote | 有（指向公开托管） | **绝不设置 remote** |

只要这两个仓从不共享一个 `.git`，公开仓里就**从来没有**真实数据可以被提交、被推送、被泄露。这是「结构上不可能」而非「靠记得别犯错」。

---

## 2. 双仓物理隔离如何杜绝误提交

隔离落在三个层面，任意一层独立生效：

### 2.1 路径隔离

- 所有真实数据只写入 `$ME_HOME/...`（默认 `~/.me`）。
- 框架文件（`SKILL.md` / `experts/` / `frameworks/` / `toolkit/`）与本仓同级，与数据根 **完全不同的目录树**。
- `scripts/init` 脚手架出的 `$ME_HOME` 会**单独 `git init`、且明确不设 remote**，是一个本地私有仓，与公开仓没有任何 git 关联。

### 2.2 公开仓 `.gitignore` 防御

即使有人误把数据文件放进公开仓目录，`.gitignore` 也会兜底拦截：

- `config/user.config` —— 只允许提交 `.example` 版本。
- `.redact-blocklist` / `*.redact-blocklist` —— 私有黑名单绝不入仓。
- `*.local` / `*.local.md` —— 任何本地个性化覆盖文件。
- `core/*.md`、`runs/`、`logs/` —— 防止有人把数据仓误链进来。
- 同时白名单保留 `templates/**` 与 `examples/**`（这些只含占位符，是安全的）。

### 2.3 占位符纪律

公开仓里任何看起来像「人」的地方，一律是占位符：`<姓名>`、`<YYYY-MM-DD>`、`{{user_name}}`。真实称呼通过 `$ME_HOME/config/user.config` 在运行时注入，永远不写进框架文件。

---

## 3. redact-check：发布前的红线 gate

`scripts/redact-check.sh` 是「最后一道闸」——命中任意红线即退出码 1，阻断发布。它有意做成**两层**，分别覆盖 CI 与本地：

### 第 1 层 · 内置通用结构模式（公开安全，CI 也能跑）

不依赖任何具体身份，因此可以安全地放进公开 CI。检测的是「结构特征」而非「具体内容」：

- 内网 IP（`10.x` / `192.168.x` / `172.16-31.x`）
- 私钥块（`-----BEGIN ... PRIVATE KEY-----`）
- 云厂商访问密钥、硬编码的 `secret/token/password/api_key`
- 会话 cookie（UUID 形态）
- 企业内网邮箱域名

这一层在没有任何私有信息的环境里也能运行，所以可以挂到公开仓的 CI 上做强制门禁。

### 第 2 层 · 私有黑名单（真名 / 公司 / 项目代号）

通用模式无法识别「真实人名」这类纯文本——它们没有结构特征。这部分交给一份**绝不入仓**的私有黑名单：

- 文件位置：`$ME_HOME/.redact-blocklist`（在私有数据仓里，不在公开仓）。
- 内容：真实人名、真实公司名、内部项目代号等，一行一条，支持 `#` 注释。
- 行为：**本地 push 前**跑 `redact-check.sh` 会自动加载它，逐条扫描公开仓工作树；一旦真名出现在待发布内容里就拦下。
- CI 上没有这个文件，所以 CI 只跑第 1 层——这正是设计意图：**真名黑名单本身就是敏感信息，绝不能上传到公开 CI。**

> 推荐把 `bash scripts/redact-check.sh` 接入本地 `pre-push` 钩子。这样「推送公开仓」这个动作天然带上了红线扫描。

## 4. 第三方隐私伦理

档案里不只有你自己，还会出现家人、同事、上级、其他干系人。**这些人没有同意被你记录、被 AI 分析。** 这带来一条硬约束：

> **凡是关于第三方的记录、判断、画像、分析，全部只能留在本地 `$ME_HOME`，永不进入任何公开或共享位置。**

具体要求：

- **数据留本地**：关于他人的 L1–L5 画像、关系笔记、冲突复盘，只写 `$ME_HOME/relations/...`，从不出仓。
- **三态标记诚实**：对第三方的描述必须用 `[事实]` / `[判断]` / `[假设]` 区分，并标 `[来源]`。不要把你的推测当成对方的客观事实记下来。
- **不贴标签**：禁止给第三方安上精神病学诊断标签或人格障碍标签。记录行为与影响，而不是给人定性。
- **对外分享前先脱敏**：如果你想把某段方法论或案例分享出去（写文章、提 issue、贡献示例），必须先抽离成占位符，并经 `redact-check` 扫过。`examples/` 里的所有案例都应是虚构或完全脱敏的。
- **最小必要**：只记录帮助你处理关系真正需要的信息，不做无关的人物档案收集。

---

## 5. 通知与后台沉淀是本地私有任务

助手的「通知」与「后台沉淀」（例如 `sleep/` 这类离线整理任务）**是你的本地私有任务，不属于本开源框架**。

- 它们运行在你的本地环境，读写的是 `$ME_HOME` 里的私有数据。
- 它们的产物（提醒、整理后的档案、夜间复盘）同样只留本地。
- 开源框架仓里**不包含、也不应包含**任何此类任务的真实配置或产出。

换句话说：框架公开你「如何思考」，但你「思考的内容」以及围绕它的自动化，全部留在 `$ME_HOME`。

---

## 一句话总结

公开的是镜头，私有的是你。两者从不共享 `.git`；`redact-check` 在发布前兜底拦截结构化密钥与私有黑名单；第三方数据无条件留本地。这套机制让隐私泄露从「需要小心避免」变成「结构上做不到」。
